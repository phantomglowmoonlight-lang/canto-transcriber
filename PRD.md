# 產品需求文檔：廣東話會議錄音轉文字系統

> 版本：v1.1
> 作者：PM
> 狀態：初稿（新增粵語→書面語翻譯、單句重新STT、降噪/提音量 toggle）

## 1. 產品定位

一套純本機運行的桌面工具，將廣東話會議錄音轉為結構化文字記錄，支援說話人分離、人名編輯、多格式匯出，並提供 AI 生成會議報告的能力。

### 1.1 目標用戶

| 用戶類型 | 使用場景 | 關鍵需求 |
|----------|----------|----------|
| 中小企業管理者 | 內部會議記錄 | 準確轉寫、識別誰說了什麼 |
| 記者/編輯 | 採訪錄音整理 | 廣東話採訪逐字稿、匯出編輯 |
| 學術研究員 | 座談會/訪談記錄 | 說話人分離、引用標記 |
| 法律/會計 | 客戶會議存檔 | 精確記錄、歸檔格式 |

### 1.2 核心價值

- 從「聽一遍錄音」到「看一眼文字」：會議記錄效率提升 5-10 倍
- 純本機運行：敏感會議內容不外洩
- 廣東話優先：專為粵語會議場景優化

## 2. 功能需求

### P0 — 必須包含（MVP）

| 編號 | 功能 | 描述 | 驗收標準 |
|------|------|------|----------|
| F-01 | 音檔載入 | 支援常見音檔格式匯入 | mp3/wav/m4a/ogg/flac/aac 至少五種；拖曳或檔案選擇器 |
| F-02 | 廣東話語音識別 | 將粵語音頻轉為繁體中文文字 | 乾淨錄音準確率 >85%（CER）；輸出帶時間戳 |
| F-03 | 長音檔自動切割 | 超過設定時長自動分段處理 | 切割點前後保留重疊（預設 30 秒）；最終輸出合併為單一連貫文本 |
| F-04 | 說話人分離 | 自動區分不同發言人 | 輸出標記為「人物 #1」「人物 #2」…；同一人物連續發言合併 |
| F-05 | 人名編輯與全篇替換 | 用戶可將代號改為真實姓名 | 修改一處即全篇同步更新；支援「人物 #1→陳大明」映射表 |
| F-06 | 粵語→書面語翻譯 | 將 STT 輸出的粵語口語翻譯為書面語 | 支援整篇翻譯；支援單句/多句選擇翻譯；保留說話人標記與時間戳；翻譯後可切換原文/書面語/對照模式檢視 |
| F-07 | TXT 匯出 | 匯出純文字格式 | 編碼 UTF-8；可選擇匯出粵語原文或書面語版本 |
| F-08 | DOCX 匯出 | 匯出帶格式 Word 文件 | 說話人姓名加粗或標色；可設定字型/字級；含時間戳；可選粵語/書面語版本 |

### P1 — 重要功能

| 編號 | 功能 | 描述 | 預期收益 |
|------|------|------|----------|
| F-09 | AI 會議報告生成 | 將轉寫文本餵給 LLM 生成結構化會議報告（議題、結論、待辦事項） | 從「逐字稿」到「會議紀要」一步到位 |
| F-10 | AI 分析可選粵語/書面語 | 生成報告時可選擇基於粵語原文或書面語翻譯版本 | 用戶可自行權衡「原汁原味」vs「正式清晰」 |
| F-11 | 單句/多句選擇重新 STT | 選取一句或多句，系統從原始音檔提取對應段落重新進行語音識別 | 局部修正識別錯誤，不用整份重跑 |

| F-14 | API 接口 | 提供 REST API 供第三方系統調用 | 可整合到現有 OA/內部系統 |
| F-15 | 進度顯示 | 轉寫過程中顯示即時進度條或百分比 | 用戶知道還有多久，減少焦慮 |
| F-16 | 音檔播放同步 | 點擊文字段落可跳到對應錄音位置 | 快速校對 / 回聽特定發言 |

## 3. 非功能需求

| 項目 | 要求 | 說明 |
|------|------|------|
| 隱私 | 完全本機，無需聯網 | 所有資料在本機處理，AI 報告可選本機 LLM（Ollama） |

| 啟動時間 | <= 30 秒 | 模型載入進度可視化 |

| 多平台 | Windows/macOS/Linux | Python 跨平台；ffmpeg 為唯一外部依賴 |

## 4. 用戶流程

### 4.1 主流程

```
┌─────────────────────────────────────────────────────────┐
│  1. 啟動程式                                              │
│     ├─ CLI: python -m app.cli transcribe 音檔.mp3        │
│     └─ WebUI: 打開瀏覽器，上傳檔案                         │
├─────────────────────────────────────────────────────────┤
│  2. 系統自動處理（無需用戶介入）                            │
│     ├─ 檢查 ffmpeg → 確認可用                             │
│     ├─ 轉換格式 → 16kHz mono WAV                         │
│     ├─ 判斷長度 → 超過 N 分鐘？自動切割                   │
│     ├─ 語音識別 → faster-whisper 轉寫（粵語）              │
│     └─ 說話人分離 → 標記人物 #1/#2/#3                    │
├─────────────────────────────────────────────────────────┤
│  3. 檢視與編輯                                            │
│     ├─ 瀏覽轉寫結果，按時間軸排列                           │
│     ├─ 看到「人物 #1」「人物 #2」…                         │
│     ├─ 輸入真實姓名 → 全篇自動替換                         │
│     └─ (可選) 翻譯成書面語 → 可切換原文/書面語/對照        │
├─────────────────────────────────────────────────────────┤
│  4. 逐句精修（可選，針對識別不準的句子）                    │
│     ├─ 選取句子 → 開啟「降噪」→ 重新 STT → 更新           │
│     ├─ 選取句子 → 開啟「提音量」→ 重新 STT → 更新         │
│     ├─ 兩者可疊加（先降噪再提音量，或反之）                │
│     └─ 選取句子 → 直接重新 STT（不加任何處理）             │
├─────────────────────────────────────────────────────────┤
│  5. 匯出                                                  │
│     ├─ TXT：可選粵語原文 / 書面語版本                      │
│     ├─ DOCX：帶格式，可選粵語原文 / 書面語版本              │
│     └─ (可選) 生成 AI 會議報告 → 可選基於粵語還是書面語    │
└─────────────────────────────────────────────────────────┘
```

### 4.2 翻譯成書面語流程

```
1. STT 完成後，每段顯示「粵語原文」
2. 用戶點擊「翻譯為書面語」按鈕（可整篇 / 選取部分）
3. 系統呼叫翻譯模型，逐句或批量處理
4. 完成後提供三種檢視模式切換：
   [原文] [書面語] [對照]
   - 原文：只顯示粵語口語轉寫
   - 書面語：只顯示翻譯後的書面語
   - 對照：上半部粵語原文，下半部書面語翻譯
5. 用戶切換模式不影響人名編輯、說話人標記等
6. 如果對某句重新 STT，對應的書面語翻譯標記為「過期」，
   提示用戶重新翻譯
```

### 4.3 逐句精修流程（降噪 / 提音量 / 重新 STT）

```
情境：用戶發現「人物 #1」某段話轉出來是亂碼或明顯錯誤

1. 用戶在轉寫結果中選取那幾句（單擊選一句，Ctrl+多選）
2. 介面出現兩個 toggle 按鈕：
   [🔊 降噪 OFF] [📢 提音量 OFF]
3. 用戶判斷原因：
   - 背景有雜音 → 開啟「降噪」
   - 說話太小聲 → 開啟「提音量」
   - 兩者都有 → 兩個都開
4. 點擊「重新識別所選段落」按鈕
5. 系統從原始音檔提取對應時間段音頻
6. 應用所選處理（降噪 / 提音量）
7. 重新跑 STT，僅更新選取的句子
8. 原有說話人標記保留，書面語翻譯標記為「需重新翻譯」

注意事項：
- 降噪和提音量是獨立 toggle，互不排斥
- 兩個都開時，先降噪後提音量（或程序員決定順序）
- 處理只影響選取段落的原始音頻副本，不破壞原始音檔
- 重新 STT 後，用戶可以再次選取並疊加更多處理
```

### 4.4 編輯人名流程（關鍵交互）

### 編輯人名流程（關鍵交互）

```
1. 系統輸出「人物 #1（發言 23 次）」「人物 #2（發言 18 次）」…
2. 用戶輸入：1 陳大明
3. 系統即時反饋：已將「人物 #1」→「陳大明」
4. 用戶輸入：2 李小華
5. 系統即時反饋：已將「人物 #2」→「李小華」
6. 用戶可隨時「預覽」替換後全文
7. 用戶確認 → 套用到全篇 → 輸出最終版
```

#### 邊界情況處理

- **用戶只改了部分人物**：未命名的人物保留原始代號
- **兩個人物誤標為同一人**：發出警告「人物 #2 和人物 #3 已被命名為相同姓名，是否合併？」
- **用戶輸入空姓名**：忽略，提示「姓名不可為空」
- **事後想改回代號**：提供「重置為預設代號」功能

### 5.2 局部重新 STT 流程

當用戶選取句子重新 STT 或套用降噪/提音量時：

```
用戶選取句子 (segments 3-5)
       │
       ▼
從原始音檔提取 audio[seg3.start : seg5.end]
       │
       ▼
┌─ 降噪 ON? ──→ 套用降噪濾波 ──┐
│                               │
└──────── 繼續 ─────────────────┘
       │
       ▼
┌─ 提音量 ON? ──→ 增益處理 ──┐
│                             │
└──────── 繼續 ───────────────┘
       │
       ▼
   重新 STT（faster-whisper）
       │
       ▼
   更新 segments 3-5 的文字
   保留原有 speaker 標記
   書面語翻譯→標記為「需重新翻譯」
```

### 5.3 任務生命週期

```
提交任務 → pending → processing → completed → (後處理) → ready
                              ↘ 局部重新STT ↙
                              ↘ failed
```

任務狀態機：

| 狀態 | 觸發 | 下一步 |
|------|------|--------|
| pending | 用戶提交音檔 | 後台消費者拾取 → processing |
| processing | 正在轉寫 | 根據進度更新百分比；完成 → completed |
| completed | 轉寫 + 說話人分離完成 | 等待用戶操作（編輯、翻譯、逐句精修） |
| ready | 用戶滿意結果，可匯出 | 匯出 / 生成報告 |
| failed | 處理過程出錯 | 返回錯誤訊息與原因 |

> 注意：completed 之後的局部重新 STT、降噪/提音量處理不改變任務狀態，
> 只更新對應 segments 的內容。

## 6. API 規範

### 6.1 端點列表

#### 基礎操作

| 方法 | 路徑 | 請求 | 回應 | 說明 |
|------|------|------|------|------|
| POST | `/v1/transcribe` | multipart: audio 文件 + diarization 參數 | `{ task_id }` | 提交轉寫 |
| GET | `/v1/tasks/{task_id}` | - | Task 完整狀態 | 查詢結果 |
| DELETE | `/v1/tasks/{task_id}` | - | `{ ok }` | 刪除任務 |
| GET | `/v1/health` | - | `{ status: "ok" }` | 健康檢查 |

#### 翻譯與編輯

| 方法 | 路徑 | 請求 | 回應 | 說明 |
|------|------|------|------|------|
| POST | `/v1/tasks/{task_id}/translate` | `{ scope: "all"|segment_ids[] }` | `{ updated_segments }` | 將選取段落翻譯為書面語 |
| PUT | `/v1/tasks/{task_id}/speakers` | `{ "人物 #1": "陳大明", ... }` | 更新後的 speaker_map | 更新人名 |

#### 逐句精修

| 方法 | 路徑 | 請求 | 回應 | 說明 |
|------|------|------|------|------|
| POST | `/v1/tasks/{task_id}/retranscribe` | `{ segment_ids: [3,4,5], noise_reduction: bool, volume_boost: bool }` | `{ updated_segments }` | 對選取段落重新 STT，可疊加降噪/提音量 |

#### 匯出與報告

| 方法 | 路徑 | 請求 | 回應 | 說明 |
|------|------|------|------|------|
| POST | `/v1/tasks/{task_id}/export` | `{ format: "txt"|"docx", language: "yue"|"written" }` | 文件下載 | 匯出結果（可選粵語原文或書面語） |
| POST | `/v1/tasks/{task_id}/report` | `{ provider, model, language: "yue"|"written", prompt_template? }` | `{ report }` | AI 會議報告（可選基於粵語或書面語） |

### 6.2 設計決策說明

- **使用 task_id 非同步模式而非同步返回**：轉寫可能需要數分鐘，同步等待不現實
- **匯出直接回傳文件而非 base64**：支援串流下載，減少記憶體開銷
- **speaker_map 獨立端點更新**：前端可在不重新轉寫的情況下迭代編輯人名
- **相容 OpenAI 格式的 LLM 整合**：一套 prompt 邏輯同時支援 Ollama 和 OpenAI
- **retranscribe 端點的冪等性**：每次重新 STT 都基於原始音檔 + 本次指定的處理參數，不累積歷史處理效果
- **translate 端點的 scope 設計**：支援 `"all"` 或 `segment_ids[]`，讓前端可以靈活控制翻譯粒度
- **降噪/提音量參數放在 retranscribe 端點而非獨立端點**：因為這兩個功能總是伴隨重新 STT 使用，分開會增加 race condition

## 7. 技術選型建議

### 7.1 核心引擎對比

#### 語音識別方案

| 方案 | 廣東話支援 | 本機運行 | 準確率 | 模型大小 | 建議 |
|------|-----------|---------|--------|---------|------|
| faster-whisper large-v3 | 極佳 | ✅ | **最高** | ~3GB | **首選**，準確率最強的開源本機方案 |
| FunASR (SenseVoice) | 佳 | ✅ | 中高 | ~1GB | 可作為備選參考 |
| Azure STT | 極佳 | ❌ 需聯網 | 高 | - | 不合本機需求 |

**結論**：faster-whisper large-v3 是目前本機廣東話語音識別準確率最高的方案。不考慮速度最佳化，使用 float16 計算精度以保留最佳辨識品質。

#### 說話人分離方案

| 方案 | 精確度 | 配置複雜度 | 建議 |
|------|--------|-----------|------|
| pyannote-audio 3.1 | 高 | 需 HF token + 同意條款 | 精準模式 |
| webrtcvad + 時間聚類 | 中 | 零配置 | 輕量模式（預設） |
| NVIDIA NeMo | 高 | 需 GPU + 複雜安裝 | 暫不考慮 |

**結論**：預設使用 VAD 模式（零門檻），提供 pyannote 模式切換（需用戶自行申請 HF token）。

#### 粵語→書面語翻譯方案

| 方案 | 本機運行 | 精確度 | 建議 |
|------|---------|--------|------|
| 本機 LLM（Ollama + Gemma/Qwen） | ✅ | 中高 | **首選**，一次部署多用途 |
| 專用翻譯 API | ❌ 需聯網 | 高 | 不合本機需求 |
| 規則式 + 詞典映射 | ✅ | 低（僅限固定句式） | 不建議，廣東話書面語轉換需理解語義 |

**結論**：利用現有 Ollama 或同機 LLM 做粵語→書面語翻譯，既可本機運行，又能與 AI 會議報告共用模型。

#### 音頻降噪方案

| 方案 | 說明 | 建議 |
|------|------|------|
| noisereduce（Python 套件） | 基於頻譜閘值的降噪 | **首選**，純 Python，無外部依賴 |
| sox（命令行工具） | 內建降噪效果器 | 備選，但需 sox 安裝 |
| ffmpeg anlmdn 濾波器 | ffmpeg 內建降噪 | 可直接利用現有 ffmpeg |

**結論**：優先使用 noisereduce 套件，ffmpeg anlmdn 作為備選。提音量使用 pydub 的增益控制即可。

### 7.2 輸出格式

| 格式 | 適用場景 | 實現方式 |
|------|---------|---------|
| TXT | 快速分享、複製貼上 | 內建文字處理 |
| DOCX | 正式歸檔、上呈報告 | python-docx：說話人加粗，時間戳灰色小字 |
| PDF（P2） | 跨平台正式文件 | 可透過 DOCX 轉換或直接生成 |

### 7.3 DOCX 格式規範建議

- 字型：Noto Sans TC / 微軟正黑體
- 說話人姓名：粗體 + 深藍色
- 時間戳：灰色較小字級，靠右或內縮（端看產品定位）
- 段落間距：1.15 倍行高
- 頁首：可設定會議名稱、日期

## 8. 邊界情況與錯誤處理

### 8.1 音檔相關

| 情況 | 系統行為 | 提示訊息 |
|------|---------|---------|
| 不支援的格式（.wmv） | 拒絕處理 | 「不支援的格式，請轉為 mp3/wav/m4a」 |
| 音檔長度 < 1 秒 | 拒絕處理 | 「音檔過短，請確認上傳正確的錄音」 |
| 音檔長度 > 10 小時 | 警告但可處理 | 「音檔較長（X 小時），預計處理時間約 Y 分鐘」 |
| 音檔損壞 | 拒絕處理 | 「無法讀取音檔，檔案可能已損壞」 |
| 純靜音的錄音 | 完成但結果為空 | 「未偵測到語音內容」 |
| ffmpeg 未安裝 | 啟動時檢測失敗 | 「需要 ffmpeg，請安裝後重試」 |

### 8.2 轉寫相關

| 情況 | 系統行為 | 提示訊息 |
|------|---------|---------|
| 錄音中有大量背景噪音 | 準確率下降，但繼續處理 | 完成後提示「背景噪音較大，建議改善錄音環境」 |
| 多人同時發言 | 依 VAD 決定主要發言者 | 「偵測到多人同時發言，部分文字可能不完整」 |
| 中英夾雜（代碼混合） | 盡量轉寫，部分英文可能誤識 | 「偵測到多語言混合」 |
| 非粵語內容（國語/英語） | 嘗試以 yue 模型處理，可能不準 | 「偵測到非粵語段落，準確率可能偏低」 |
| 長時間（>30 秒）無發言段落 | 自動合併為「...（長時間靜音）」 | 保留時間戳標記 |

### 8.3 翻譯成書面語

| 情況 | 系統行為 |
|------|---------|
| 翻譯整篇會議記錄 | 後台批量處理，顯示進度條 |
| 只翻譯選取的句子 | 僅處理選取段落，其餘保持 null |
| 某句重新 STT 後，書面語未更新 | 自動標記 `translation_stale: true`，在 UI 顯示「需重新翻譯」標記 |
| 翻譯模型不支援的語言 | 提示「目前只支援粵語→繁體中文書面語」 |
| 翻譯結果與原文差異過大 | 提供「對照模式」方便用戶比對 |

### 8.4 逐句精修（降噪 / 提音量 / 重新 STT）

| 情況 | 系統行為 |
|------|---------|
| 用戶選取句子但未開啟任何 toggle | 直接重新 STT，不套用音頻處理 |
| 選取範圍橫跨多個說話人 | 正常處理，保留原 speaker 標記 |
| 重新 STT 後結果比原來更差 | 提供「復原為上一個版本」按鈕，保留最近 N 次歷史 |
| 降噪開啟但音檔本身很乾淨 | 降噪效果不明顯，但不影響正常使用 |
| 提音量開太大導致 clipping（削波） | 限制最大增益 12dB，避免音頻失真 |
| 選取段落太短（< 0.5 秒） | 提示「選取段落太短，可能無法有效識別」 |
| 降噪 + 提音量同時開啟 | 先降噪後提音量（或視實作決定順序），效果疊加 |
| 多次疊加降噪/提音量 | 每次重新 STT 都基於原始音檔，不累積疊加，避免音質惡化 |

### 8.6 人名編輯

| 情況 | 系統行為 |
|------|---------|
| 兩個人物被改了相同名字 | 提示「人物 #2 和 #3 都被命名為『陳大明』，是否合併？」 |
| 僅部分人物被命名 | 未命名的保持「人物 #N」代號 |
| 命名後又改回 | 支援任意的重新命名，以最後一次為準 |
| 人物數量 > 10 人 | 正常處理，UI 需要支援滾動查看 |

### 8.7 資源限制

| 情況 | 系統行為 |
|------|---------|
| 記憶體不足（模型載入失敗） | 提示用戶切換小模型（medium） |
| 硬碟空間不足（寫入輸出） | 明確提示剩餘空間與所需空間 |
| GPU 記憶體不足 | 自動 fallback 到 CPU |
| 模型下載失敗（首次啟動） | 提示檢查網絡，支援手動下載模型 |

## 9. 測試要點

### 9.1 功能測試

| 測試案例 | 預期結果 |
|---------|---------|
| 上傳 10 分鐘乾淨粵語錄音 | 轉寫完成，字元錯誤率（CER）< 10% |
| 上傳 30 分鐘錄音（>10 分鐘閾值） | 自動切割，最終合併結果連貫無重複 |
| 多人會議錄音 | 正確標記至少 2 個不同說話人 |
| 修改說話人名稱 | 全文即時更新，匯出文件使用新名稱 |
| 匯出 DOCX | 文件格式正確，說話人粗體，時間戳可見 |

#### 新增功能測試

| 測試案例 | 預期結果 |
|---------|---------|
| 整篇翻譯為書面語 | 粵語口語轉為書面語，保留說話人標記 |
| 切換原文/書面語/對照模式 | 三種模式顯示正確，切換流暢無延遲 |
| 選取單句重新 STT | 只更新該句，其他句子不受影響 |
| 選取多句重新 STT | 選取範圍內句子全部更新，邊界銜接自然 |
| 開啟降噪後重新 STT | 背景噪音降低，識別文字改善 |
| 開啟提音量後重新 STT | 音量提升，識別文字改善 |
| 降噪 + 提音量疊加 | 兩種處理同時生效，效果優於單獨使用 |
| AI 報告選擇基於書面語 | 報告語言更正式，保留原始會議結構 |
| AI 報告選擇基於粵語原文 | 報告保留口語語氣和原話引用 |
| 重新 STT 後書面語標記為過期 | 對應句子的 `translation_stale: true`，UI 顯示提示 |

### 9.2 邊界測試

| 測試案例 | 預期結果 |
|---------|---------|
| 空音檔（全靜音） | 提示「未偵測到語音」 |
| 極長音檔（5 小時） | 正常切割處理，合併結果完整 |
| 惡意檔案（exe 偽裝為 mp3） | 優雅拒絕，不當機 |
| 同時提交 10 個任務 | 正確排隊，不丟失任務 |

### 9.3 驗收標準

1. 一段 30 分鐘、2 人對話的乾淨粵語錄音，轉寫結果字元錯誤率（CER）< 10%
2. 說話人標記準確率 >70%
3. 人名修改後，匯出文件中所有對應發言均使用新名稱
4. API 服務器重啟後，未完成的任務恢復或明確丟棄
